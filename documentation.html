Gym Management System Documentation

1. Application Lifecycle

graph TD A[Application Start] --> B[GymManagementSystem.java] B --> C[Load FXMLDocument.fxml] C --> D[Initialize Login Screen] D --> E[User Authentication] E --> F[Load Dashboard.fxml] F --> G[Initialize Dashboard Controller] G --> H[Application Running]

The application follows a typical JavaFX lifecycle:

  • The application starts with GymManagementSystem.java as the entry point
  • Initial FXML document is loaded (login screen)
  • After successful authentication, the dashboard is loaded
  • The application remains running until explicitly closed

2. Component Interaction

graph LR A[FXML Views] --> B[Controllers] B --> C[Data Models] C --> D[Database] B --> E[CSS Styling] A --> E

Key components and their interactions:

  • FXML Views: Define the UI layout (FXMLDocument.fxml, dashboard.fxml)
  • Controllers: Handle user interactions (FXMLDocumentController.java, dashboardController.java)
  • Data Models: Manage data (memberData.java, coachData.java)
  • Database: Stores persistent data (database.java)
  • CSS: Handles styling (loginDesign.css, dashboardDesign.css)

3. Database Interaction

graph TD A[Java Classes] --> B[database.java] B --> C[SQL Database] D[memberData.java] --> B E[coachData.java] --> B

Database interaction flow:

  • The database.java class serves as the database connector
  • Data model classes (memberData.java, coachData.java) use the database class to perform CRUD operations
  • SQL queries are executed through JDBC (Java Database Connectivity)
  • Results are mapped to Java objects for use in the application

4. UI and Core Code Interaction

graph TD A[FXML Views] --> B[Controllers] B --> C[Event Handlers] C --> D[Data Operations] D --> E[UI Updates] E --> A

UI and core code interaction:

  • FXML files define the UI structure
  • Controllers contain event handlers for user interactions
  • Event handlers trigger data operations
  • Results are reflected back in the UI through JavaFX properties and bindings

5. Authentication and Roles

graph TD A[Login Screen] --> B{Authentication} B -->|Success| C[Role Check] C -->|Admin| D[Admin Dashboard] C -->|Member| E[Member Dashboard] C -->|Coach| F[Coach Dashboard] B -->|Failure| A

Authentication and role management:

  • The system uses a login-based authentication system
  • Roles are managed through the database
  • Different dashboards and permissions are provided based on user roles
  • Session management is handled through the application controllers

Key Differences from Android/Kotlin

  • JavaFX uses FXML instead of XML layouts
  • Controllers are explicitly defined and linked to FXML files
  • Event handling is done through Java methods rather than Kotlin lambdas
  • UI updates are managed through JavaFX properties instead of LiveData
  • Database operations use JDBC instead of Room
